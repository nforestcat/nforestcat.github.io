서비스 기획, API, 백엔드, DB 기초 정리

5월 27, 2026

들어가며

서비스 기획을 공부하면서 단순히 “사용자 화면”만 생각하는 것이 아니라, 관리자 페이지, API, DB, 보안, 법적 이슈, 배포, 포트폴리오까지 함께 고려해야 한다는 점을 정리했다.

예시 서비스로는 알러지 정보를 알려주는 음식점 지도 앱을 기준으로 생각했다.

핵심 아이디어는 다음과 같다.

사용자가 위치 정보를 켜고 앱에 들어가면, 주변 음식점의 어떤 메뉴에 어떤 알러지 성분이 들어 있는지 알려주는 지도 서비스

이 서비스는 단순 맛집 앱이 아니라 위치 기반 서비스, 음식 데이터 플랫폼, 사용자 제보 시스템, 개인정보 및 위치정보 처리까지 포함하는 서비스다.

1. 서비스 기획의 본질

서비스 기획의 본질은 사용자를 덜 귀찮게 만드는 것이다.

좋은 서비스는 사용자가 반복해서 입력하거나 직접 판단해야 하는 일을 줄여준다.

좋은 서비스 기획 = 사용자의 귀찮음을 회사가 대신 떠안는 구조

알러지 지도 앱 기준으로는 사용자가 매번 알러지 정보를 입력하면 안 된다.

나쁜 UX는 다음과 같다.

앱 실행
→ 위치 입력
→ 알러지 다시 선택
→ 음식점 검색
→ 메뉴 하나하나 확인
→ 위험 성분 직접 판단

좋은 UX는 다음과 같다.

최초 1회 알러지 설정
→ 이후 앱 실행 시 자동 적용
→ 주변 음식점 위험도 자동 표시

즉, 사용자는 한 번만 입력하고, 이후의 판단은 서비스가 자동으로 처리해야 한다.

2. 알러지 지도 앱 서비스 개요

서비스 목적

음식 알러지가 있는 사용자가 주변 음식점의 메뉴별 알러지 정보를 확인하고, 위험 메뉴를 사전에 피할 수 있도록 도와주는 위치 기반 음식점 정보 서비스다.

주요 사용자

사용자 설명
음식 알러지 보유자 특정 성분 회피 필요
부모 아이의 식단 안전 확인
비건/종교식 사용자 특정 재료 회피
음식점 관리자 메뉴 및 원재료 정보 관리
서비스 운영자 신고, 검수, 데이터 품질 관리

핵심 기능

구분 기능
일반 사용자 위치 기반 음식점 조회
일반 사용자 알러지 필터 설정
일반 사용자 메뉴별 위험도 확인
일반 사용자 허위 정보 신고
음식점 관리자 메뉴 등록 및 수정
음식점 관리자 알러지 성분 등록
운영 관리자 신고 검수
운영 관리자 사용자 및 데이터 관리

3. 초보 기획자와 고수 기획자의 차이

초보 기획자는 일반 사용자의 화면과 기능만 생각하기 쉽다.

예를 들면 다음 정도만 생각한다.

사용자가 앱을 연다
→ 음식점을 본다
→ 알러지 정보를 확인한다

하지만 실무에서는 이것만으로 부족하다.

고수 기획자는 다음까지 함께 고려한다.

관리자 페이지는 필요한가?
허위 정보는 어떻게 처리할 것인가?
음식점 데이터는 누가 입력할 것인가?
개인정보와 위치정보는 어떻게 처리할 것인가?
알러지 정보가 틀렸을 때 책임은 어떻게 되는가?
DB 구조는 어떻게 설계할 것인가?
API는 어떤 단위로 만들 것인가?

즉, 서비스 기획은 화면 기획만이 아니라 운영, 데이터, 보안, 법적 이슈까지 포함한다.

4. 플로우차트란?

플로우차트는 서비스나 프로그램의 흐름을 도형과 화살표로 시각화한 것이다.

쉽게 말하면 다음을 한눈에 보여주는 도구다.

사용자가 무엇을 하고
시스템이 어떻게 반응하고
어떤 조건에서 다음 단계로 넘어가는지

플로우차트를 작성하면 예외 상황을 발견하기 쉽다.

예를 들어 알러지 지도 앱에서는 다음과 같은 예외를 생각해야 한다.

위치 권한을 거부하면?
알러지 정보를 설정하지 않았으면?
주변 음식점 데이터가 없으면?
메뉴 알러지 정보가 없으면?
허위 신고가 들어오면?

5. 알러지 지도 앱 사용자 플로우차트

flowchart TD
    A([앱 실행]) --> B{로그인 상태인가?}

    B -- 아니오 --> C[로그인 / 회원가입]
    C --> D[알러지 정보 설정]

    B -- 예 --> E{알러지 정보가 등록되어 있는가?}
    E -- 아니오 --> D
    E -- 예 --> F[위치 권한 요청]

    D --> F

    F --> G{위치 권한 허용?}
    G -- 아니오 --> H[위치 권한 필요 안내]
    H --> I[수동 지역 검색 제공]

    G -- 예 --> J[현재 위치 수집]
    I --> K[음식점 검색]
    J --> K[주변 음식점 조회]

    K --> L{주변 음식점 데이터 있음?}
    L -- 아니오 --> M[주변 음식점 없음 안내]
    L -- 예 --> N[사용자 알러지 정보와 메뉴 성분 비교]

    N --> O[음식점별 위험도 계산]
    O --> P[지도에 음식점 표시]

    P --> Q[음식점 선택]
    Q --> R[메뉴별 알러지 정보 표시]

    R --> S{정보가 잘못되었는가?}
    S -- 아니오 --> T[즐겨찾기 / 길찾기 / 공유]
    S -- 예 --> U[허위 정보 신고 작성]

    U --> V[신고 접수]
    V --> W[관리자 검수 대기]

    T --> X([사용 종료])
    W --> X

6. 기획 플로우

서비스 기획은 보통 다음 흐름으로 진행된다.

서비스 아이디어 정리
→ 사용자 플로우차트 작성
→ Figma로 화면 설계
→ 화면 페이지 목록 작성
→ 각 화면에서 필요한 데이터 정리
→ API 설계서 작성
→ DB 설계
→ 개발
→ 테스트
→ 배포

요약하면 다음과 같다.

플로우차트로 사용자 흐름을 정리하고,
Figma로 화면을 만들고,
각 화면에서 필요한 데이터를 정리한 뒤,
그 데이터를 주고받는 API를 설계하고,
API 설계서를 기준으로 개발한다.

7. Figma란?

Figma는 앱이나 웹 화면을 설계하는 도구다.

서비스 개발 전에 다음과 같은 화면을 미리 그릴 수 있다.

홈 지도 화면
음식점 상세 화면
알러지 설정 화면
로그인 화면
관리자 페이지

Figma는 단순한 그림 도구가 아니라 기획자, 디자이너, 개발자가 함께 협업하는 도구다.

실무 흐름은 보통 다음과 같다.

서비스 기획
→ Figma 화면 설계
→ 개발자 전달
→ 실제 개발

알러지 지도 앱 기준으로는 최소한 다음 화면이 필요하다.

화면 설명
온보딩 화면 서비스 소개
로그인 화면 회원가입/로그인
알러지 설정 화면 계란, 우유, 땅콩 등 선택
위치 권한 안내 화면 위치 사용 이유 설명
지도 홈 화면 주변 음식점 표시
음식점 상세 화면 메뉴별 알러지 정보 표시
신고 화면 잘못된 정보 신고
마이페이지 내 알러지 정보 관리

8. 화면 기반 API 설계

화면을 설계한 뒤에는 각 화면에서 필요한 데이터를 정리해야 한다.

예를 들어 지도 홈 화면에는 다음 데이터가 필요하다.

음식점명
위치
거리
위험도
사용자 알러지와 매칭된 성분

이 데이터를 가져오기 위해 API가 필요하다.

GET /api/restaurants/nearby?lat=37.5665&lng=126.9780&radius=1000

응답 예시는 다음과 같다.

{
  "restaurants": [
    {
      "restaurant_id": 101,
      "name": "서울분식",
      "distance": 230,
      "risk_level": "warning",
      "matched_allergies": ["egg", "milk"]
    }
  ]
}

화면과 API를 연결하면 다음과 같다.

화면 필요한 API
로그인 화면 POST /api/auth/login
알러지 설정 화면 GET /api/allergens, POST /api/users/me/allergies
지도 홈 화면 GET /api/restaurants/nearby
음식점 상세 화면 GET /api/restaurants/{id}
신고 화면 POST /api/reports
마이페이지 GET /api/users/me, PATCH /api/users/me

9. REST API란?

REST API는 웹에서 가장 많이 사용하는 API 설계 방식이다.

쉽게 말하면 다음과 같다.

URL과 HTTP 규칙으로 데이터를 주고받는 방식

REST API는 자원을 중심으로 설계한다.

알러지 지도 앱 기준 자원은 다음과 같다.

자원 의미
users 사용자
restaurants 음식점
menus 메뉴
reports 신고

HTTP Method는 다음처럼 사용한다.

Method 의미
GET 조회
POST 생성
PUT 전체 수정
PATCH 일부 수정
DELETE 삭제

예를 들어 음식점 관련 API는 다음과 같이 설계할 수 있다.

GET    /restaurants       음식점 목록 조회
GET    /restaurants/101   101번 음식점 조회
POST   /restaurants       음식점 생성
PATCH  /restaurants/101   101번 음식점 수정
DELETE /restaurants/101   101번 음식점 삭제

10. JSON과 XML의 차이

API에서 데이터를 주고받을 때 대표적인 형식이 JSON과 XML이다.

요즘 웹/앱 서비스에서는 대부분 JSON을 사용한다.

JSON 예시

{
  "name": "Kim",
  "phone": "010-1234-5678",
  "allergy": ["milk", "peanut"]
}

XML 예시

<user>
    <name>Kim</name>
    <phone>010-1234-5678</phone>
    <allergy>milk</allergy>
    <allergy>peanut</allergy>
</user>

비교하면 다음과 같다.

항목 JSON XML
가독성 좋음 상대적으로 복잡
데이터 크기 작음 큼
처리 속도 빠름 상대적으로 느림
모바일 최적화 좋음 불리
주 사용 분야 웹/앱 API 공공, 금융, 레거시 시스템

알러지 지도 앱 같은 서비스는 일반적으로 JSON 기반 API가 적합하다.

11. 클라이언트, 웹 서버, 백엔드, 데이터베이스 아키텍처

전체 구조는 다음과 같다.

[클라이언트]
모바일 앱 / 웹 브라우저
        ↓
[웹 서버]
HTTPS / 정적 파일 제공 / 라우팅
        ↓
[백엔드 서버]
API 처리 / 인증 / 위치 검색 / 알러지 매칭
        ↓
[데이터베이스]
사용자 / 음식점 / 메뉴 / 알러지 / 신고 데이터 저장

각 역할은 다음과 같다.

구성 요소 역할
클라이언트 사용자가 보는 화면, 위치 수집, API 요청
웹 서버 HTTPS 처리, 요청 라우팅, CORS, 트래픽 제어
백엔드 인증, 비즈니스 로직, DB 조회, JSON 응답
데이터베이스 서비스 데이터 저장

12. 백엔드란?

백엔드는 사용자에게 직접 보이지 않는 서버 쪽 로직과 데이터 처리를 담당한다.

쉽게 말하면 다음과 같다.

앱 뒤에서 실제 일을 하는 부분

알러지 지도 앱에서 사용자가 현재 위치 기반 음식점을 요청하면 백엔드는 다음 작업을 한다.

1. 사용자 인증 확인
2. 위치 좌표 확인
3. DB에서 주변 음식점 조회
4. 사용자 알러지 정보 조회
5. 메뉴 성분과 비교
6. 위험도 계산
7. JSON 응답 반환

백엔드 주요 모듈은 다음과 같이 나눌 수 있다.

모듈 역할
Auth 로그인/회원가입
User 사용자 정보 및 알러지 프로필 관리
Restaurant 음식점 정보 조회
Menu 메뉴 및 성분 관리
Risk Engine 알러지 위험도 계산
Report 허위 정보 신고 처리
Admin 관리자 기능

13. 동기 처리와 비동기 처리

서비스 설계에서는 어떤 작업을 동기로 처리하고 어떤 작업을 비동기로 처리할지 정해야 한다.

동기 처리

동기 처리는 요청한 작업이 끝날 때까지 기다리는 방식이다.

요청
→ 처리 완료
→ 응답 반환

즉시 결과가 필요한 기능에 사용한다.

알러지 지도 앱에서 동기 처리할 기능은 다음과 같다.

기능 이유
로그인 즉시 성공/실패를 알아야 함
주변 음식점 조회 지도에 바로 표시해야 함
음식점 상세 조회 화면 전환 시 즉시 필요
알러지 설정 저장 저장 완료 여부 필요

비동기 처리

비동기 처리는 요청을 먼저 접수하고, 실제 처리는 뒤에서 진행하는 방식이다.

요청
→ 접수 완료 응답
→ 백그라운드 처리
→ 나중에 결과 반영

알러지 지도 앱에서 비동기 처리할 기능은 다음과 같다.

기능 이유
신고 검수 즉시 처리할 필요 없음
알림 발송 외부 서비스 의존
로그 저장 매 요청마다 동기 저장하면 느려짐
통계 계산 무거운 작업
AI 메뉴 분석 시간이 오래 걸릴 수 있음
OCR 분석 처리 시간이 길 수 있음

비동기 처리를 위해 Queue를 사용할 수 있다.

사용자 신고
→ API 서버가 신고 접수
→ Queue 등록
→ Worker가 백그라운드 처리
→ DB 반영

14. Database CRUD

CRUD는 데이터베이스에서 데이터를 다루는 기본 4가지 작업이다.

C = Create   생성
R = Read     조회
U = Update   수정
D = Delete   삭제
CRUD 의미 SQL 예시 서비스 예시
Create 새 데이터 생성 INSERT 회원가입, 음식점 등록
Read 데이터 조회 SELECT 음식점 목록 보기
Update 기존 데이터 수정 UPDATE 알러지 정보 수정
Delete 데이터 삭제 DELETE 신고 삭제, 계정 탈퇴

REST API와 CRUD는 거의 1:1로 연결된다.

CRUD HTTP Method API 예시
Create POST POST /restaurants
Read GET GET /restaurants/101
Update PUT/PATCH PATCH /restaurants/101
Delete DELETE DELETE /restaurants/101

실무에서는 데이터를 완전히 지우는 물리 삭제보다 삭제 표시만 남기는 논리 삭제를 많이 사용한다.

UPDATE users
SET deleted_at = NOW()
WHERE user_id = 1;

15. RDB, NoSQL, VectorDB 차이

데이터베이스는 목적에 따라 여러 종류가 있다.

한 줄로 정리하면 다음과 같다.

RDB      = 정리된 표 데이터
NoSQL    = 유연한 대용량 데이터
VectorDB = AI 의미 검색 데이터

RDB

RDB는 관계형 데이터베이스다.

대표 예시는 다음과 같다.

MySQL
PostgreSQL
Oracle
MariaDB
MS SQL Server

정형 데이터에 강하다.

예를 들어 다음 데이터는 RDB에 적합하다.

사용자
음식점
메뉴
알러지 성분
신고
관리자 권한

NoSQL

NoSQL은 비정형 또는 반정형 데이터에 적합하다.

대표 예시는 다음과 같다.

MongoDB
Redis
DynamoDB
Firebase Firestore
Elasticsearch

NoSQL은 다음 용도로 많이 사용된다.

로그

JSON 문서
실시간 상태값

이미지 자체는 보통 DB가 아니라 S3 같은 Object Storage에 저장하고, DB에는 이미지 URL이나 메타데이터를 저장한다.

VectorDB

VectorDB는 AI/LLM 프로젝트에서 자주 사용하는 데이터베이스다.

16. ChromaDB란?

ChromaDB는 VectorDB의 한 종류다.

주로 LLM, RAG, 문서 검색, 의미 기반 검색에 사용된다.

일반적인 흐름은 다음과 같다.

문서 업로드
→ 문서를 작은 조각으로 나눔
→ 각 조각을 embedding으로 변환
→ ChromaDB에 저장
→ 질문이 들어오면 관련 문서 조각 검색
→ LLM이 답변 생성

알러지 지도 앱 기준으로는 다음처럼 사용할 수 있다.

PostgreSQL:
사용자, 음식점, 메뉴, 알러지 성분 저장

ChromaDB:
메뉴 설명, 리뷰, 원재료 문서의 의미 검색/AI 추천

17. Firebase와 Supabase

Firebase와 Supabase는 모두 BaaS다.

BaaS는 Backend as a Service의 약자로, 직접 서버와 DB, 인증 기능을 전부 만들지 않고 미리 제공되는 백엔드 기능을 사용하는 방식이다.

프론트엔드/앱
→ Firebase 또는 Supabase
→ DB, 로그인, 파일 저장, API 기능 사용

Firebase

Firebase는 Google이 제공하는 앱/웹 개발 플랫폼이다.

주요 기능은 다음과 같다.

기능 설명
Authentication 로그인/회원가입
Firestore NoSQL 데이터베이스
Realtime Database 실시간 DB
Cloud Storage 이미지/파일 저장
Hosting 웹 배포
Cloud Functions 서버리스 함수
Analytics 사용자 행동 분석
Cloud Messaging 푸시 알림

Firebase는 NoSQL 기반 앱을 빠르게 만들 때 유리하다.

Supabase

Supabase는 PostgreSQL 기반의 Firebase 대안으로 자주 언급된다.

주요 기능은 다음과 같다.

기능 설명
PostgreSQL DB 관계형 데이터베이스
Auth 로그인/회원가입
Storage 이미지/파일 저장
Realtime 실시간 데이터 반영
Auto API DB 기반 API 자동 생성
Edge Functions 서버리스 함수
Row Level Security 행 단위 권한 제어

알러지 지도 앱은 사용자, 음식점, 메뉴, 알러지 성분, 신고 등 관계가 명확하므로 Supabase/PostgreSQL이 더 자연스러울 가능성이 높다.

18. Firebase와 Supabase의 쿼리 차이

Firebase와 Supabase는 데이터베이스 성격이 다르기 때문에 쿼리 방식도 다르다.

Firebase = 문서(Document)를 가져오는 방식
Supabase = SQL/RDB 테이블을 조회하는 방식

Firebase 예시

const q = query(
  collection(db, "restaurants"),
  where("risk_level", "==", "warning")
);

const snapshot = await getDocs(q);

Supabase 예시

const { data, error } = await supabase
  .from("restaurants")
  .select("*")
  .eq("risk_level", "warning");

SQL로 보면 다음과 같다.

SELECT *
FROM restaurants
WHERE risk_level = 'warning';

관계가 복잡한 서비스는 Supabase/PostgreSQL 쪽이 더 자연스러운 경우가 많다.

19. PL/SQL이란?

PL/SQL은 Oracle 데이터베이스에서 사용하는 프로그래밍 언어다.

정식 이름은 다음과 같다.

Procedural Language / SQL

쉽게 말하면 SQL에 프로그래밍 기능을 추가한 Oracle 전용 언어다.

SQL은 데이터를 조회하거나 수정하는 언어다.

SELECT *
FROM users
WHERE user_id = 1;

PL/SQL은 여기에 변수, 조건문, 반복문, 예외 처리, 함수, 프로시저 등을 추가한다.

DECLARE
    v_user_count NUMBER;
BEGIN
    SELECT COUNT(*)
    INTO v_user_count
    FROM users;

    IF v_user_count > 1000 THEN
        DBMS_OUTPUT.PUT_LINE('사용자가 1000명 이상입니다.');
    ELSE
        DBMS_OUTPUT.PUT_LINE('사용자가 1000명 미만입니다.');
    END IF;
END;

PL/SQL은 은행, 보험, 공공기관, 대기업 ERP, 회계 시스템 등 Oracle 기반 시스템에서 많이 사용된다.

21. 웹크롤링보다 API가 강조되는 이유

예전에는 데이터 수집 방법으로 웹크롤링을 많이 다뤘지만, 요즘은 API를 통한 데이터 수집이 더 강조된다.

차이는 다음과 같다.

크롤링 = 남의 웹페이지를 긁어오는 방식
API = 데이터 제공자가 정해둔 공식 통로로 가져오는 방식

비교하면 다음과 같다.

구분 웹크롤링 API
방식 웹페이지 HTML을 긁음 공식 데이터 통로 사용
안정성 낮음 높음
법적 안정성 주의 필요 상대적으로 안정적
데이터 형식 HTML 중심 JSON/XML 중심
유지보수 어려움 쉬움
차단 가능성 높음 정책 범위 내 사용
실무 활용 제한적 매우 많음

알러지 지도 앱에서는 데이터 수집 방식을 다음 순서로 고려하는 것이 좋다.

공식 API 우선
→ 제휴 데이터
→ 관리자 직접 입력
→ 사용자 제보
→ 필요 시 제한적 크롤링 검토

22. API 문서가 중요한 이유

서비스가 성장하면 데이터가 쌓인다.

데이터가 쌓이면 단순 앱이 아니라 플랫폼이 될 수 있다.

서비스 출시
→ 사용자 증가
→ 데이터 축적
→ 데이터 가치 발생
→ API로 외부에 제공
→ 새로운 수익모델 가능

예를 들어 알러지 지도 앱이 성장하면 다음 데이터를 보유하게 된다.

데이터 예시
음식점 데이터 위치, 영업시간, 메뉴
메뉴 데이터 메뉴명, 가격, 성분
알러지 데이터 계란 포함, 우유 포함, 땅콩 위험
사용자 행동 데이터 어떤 메뉴를 많이 조회하는지
신고 데이터 어떤 정보가 자주 틀리는지
안전 음식점 데이터 알러지 사용자들이 신뢰하는 음식점

이 데이터를 API로 외부 서비스에 제공할 수 있다.

예를 들어 배달앱이 API를 호출해 메뉴별 알러지 정보를 가져갈 수 있다.

GET /api/v1/restaurants/101/menus/allergens

응답 예시는 다음과 같다.

{
  "restaurant_id": 101,
  "restaurant_name": "서울분식",
  "menus": [
    {
      "menu_name": "김밥",
      "allergens": ["egg", "sesame"],
      "risk_level": "warning"
    }
  ]
}

즉, API 문서는 단순 개발 문서가 아니라 데이터 상품의 설명서가 될 수 있다.

28. 오픈소스에서 문서화가 중요한 이유

오픈소스를 많이 활용하도록 도와주는 핵심은 문서화다.

코드가 좋아도 문서가 없으면 대부분의 사람은 쓰지 않는다.

좋은 코드 + 나쁜 문서 = 사용하기 어려운 프로젝트
평범한 코드 + 좋은 문서 = 사람들이 써볼 수 있는 프로젝트
좋은 코드 + 좋은 문서 = 성장 가능한 오픈소스

기여 방법 PR, issue 안내

29. 알러지 지도 앱 포트폴리오 구성 예시

마무리

이번 정리의 핵심은 다음과 같다.

서비스 기획은 화면만 만드는 것이 아니다.
사용자 흐름, 관리자 기능, API, DB, 보안, 법적 이슈, 배포, 문서화까지 연결해서 봐야 한다.

알러지 지도 앱 예시를 통해 알 수 있는 점은 다음과 같다.

사용자는 귀찮으면 안 된다.
입력은 1회면 충분하다.
복잡한 판단은 시스템이 해야 한다.
법적 문제와 데이터 책임은 회사가 준비해야 한다.
API는 서비스 기능이면서 동시에 데이터 비즈니스의 통로가 될 수 있다.
개발자 프로젝트는 결과물뿐 아니라 문서화와 재사용성이 중요하다.

결국 좋은 서비스는 단순히 작동하는 앱이 아니라, 사용자가 편하고, 운영자가 관리할 수 있으며, 개발자가 확장할 수 있고, 외부에서도 활용 가능한 구조를 갖춘 서비스다.
